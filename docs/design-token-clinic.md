# Design: Token Clinic — Pre-Flight Gate for LLM Coding Spend

Generated by /office-hours on 2026-06-25
Branch: main
Repo: github.com/mrdulasolutions/TokenClinic
Status: DRAFT
Mode: Startup

## Problem Statement

Teams running LLM coding agents are paying premium-model prices for work that
cheap local tools, or a competent human, would resolve instantly. The founder's
own dev team burns thousands of dollars a month on agent calls, a meaningful but
unmeasured fraction of which is spent on trivial findings (missing imports,
lint-class issues, things "a real dev would point out in one second").

Token Clinic is a pre-flight gate: it runs cheap local analysis before any LLM
touches the code, eliminates what it can for $0, routes only the irreducible work
to the right-priced model, verifies the fix, and prints a bill showing the savings.
Thesis: don't pay Opus to find a missing import.

## Demand Evidence

- **Strongest evidence (witnessed, first-person):** The founder's own dev team
  spends "thousands a month" on LLM agent calls, and the founder has run audits
  showing models burning tokens on reasoning a competent dev would settle in one
  second. This is real, witnessed pain on a real budget. Sample size: one
  organization (their own).
- **Weak / discounted evidence:** "Enterprises worried about token budgets" and
  the Uber 500M-token-burn headline. These prove the category has money. They are
  not demand for *this* product, and no specific enterprise champion has been
  identified or spoken to.
- **Honest gap:** The core quantity, *what fraction of spend is eliminable-class
  waste*, is assumed from informal audits, not measured. The entire business rides
  on this fraction being large. It has not been computed from real log data.

## Status Quo (the real competitor)

What people do today instead of Token Clinic:
- Eat the cost.
- Manually set a cheaper default model and bump to a premium model by hand. This
  is the cheap substitute that captures maybe ~70% of the savings for $0 and zero
  install. **Its weakness: discipline decays. People forget to switch.** Token
  Clinic's automated routing removes that friction.
- Set token/spend caps, or turn agents off on large repos.

Token Clinic must beat "just default to a cheaper model and escalate by hand."
Its edge over that substitute is (a) automated routing so humans can't forget, and
(b) the differentiator the substitute cannot do: **eliminating findings locally for
$0 before any model runs.**

## Target User & Narrowest Wedge

- **v1 user:** Individual developers and small team leads watching their own
  Anthropic/LLM bill climb. The person who `npm install`s a CLI in 30 seconds and
  feels a $200/month saving as a screaming win.
- **NOT the v1 user:** Enterprise procurement. Reasons below (see Premises 3 + the
  security note). Enterprise is the *expansion*, reached by aggregating per-dev
  receipts into an org-level number, not the entry point.
- **Wedge:** Paid bottom-up. The product's own output (the EOB receipt) is the
  monetization mechanism: it proves its ROI in dollars on every run, making paid
  conversion trivial ("saved you $40, costs $15").
- **Founder motivation (named explicitly):** wants revenue, "tired of shipping free
  shit." Reframed: bottom-up does not mean free. Charge from day one; the receipt
  justifies the price. The mistake is equating "bottom-up" with "open-source toy."

## Constraints

- Pre-product. No repo, no shipped artifact, no paying customer yet.
- Solo/small effort. Cannot survive a 6-12 month enterprise sales cycle.
- The product reads source code and routes parts of it to external LLM APIs, which
  is adversarial to enterprise security review (code exfiltration). This makes
  bottom-up (own code, own machine, own API key) the low-friction path.

## Premises (all agreed by founder)

1. **The moat is local elimination, not model routing.** Routing to a cheaper model
   is table stakes and a likely future built-in of the harnesses. Killing findings
   for $0 + a compounding Health Record is the defensible part. Lead with elimination.
2. **The core thesis is unverified and must be measured before building v2.** The
   eliminable-class fraction is assumed, not quantified. Measurable this week from
   existing logs, zero code.
3. **The wedge is paid bottom-up (individual devs / small team leads), monetized by
   the self-proving receipt. Enterprise is a later expansion, not the entry point.**
4. **Distribution is a standalone CLI the founder fully controls, not an integration
   into harnesses (Claude Code, Cursor) they don't own.** Auto-switching the host
   harness is a v2+ bet on surfaces outside their control.
5. **The "$0 forever" amortization moat only holds for findings reducible to a
   deterministic rule + fixtures.** Tacit, senior-judgment findings can be routed
   cheaper but not eliminated. Two buckets, two economics. Don't promise bucket-1
   economics for bucket-2 problems.

## Approaches Considered

### Approach A: The Retroactive Audit (CHOSEN as first move)
Build the cheapest thing that produces the receipt from logs that already exist.
`tokenclinic audit` ingests a team's past LLM call logs / agent traces and prints
the EOB backwards: "last month you spent $X, here is the eliminable-class fraction,
here is what you'd have saved." Sell the audit itself.
- Effort: S. Risk: Low.
- Validates Premise 2 with a real customer's real data, earns revenue immediately,
  no autofix risk, no harness control, no security review (runs on exported logs).

### Approach B: The Standalone Clinic CLI (CHOSEN as second move)
`tokenclinic scan`, one language: Triage + local autofix + single-model escalation
+ verify + EOB. Subscription priced against demonstrated savings.
- Effort: L. Risk: Medium.
- The real recurring product. Self-controlled distribution. First runs double as the
  thesis measurement. Risk: bigger build, fix-verify loop has real edge cases, could
  ship and find savings are small.

### Approach C: Open-Core, sell the Health Record (LATER pricing model)
Triage + autofix + receipt is free and acts as the growth engine (every receipt is
an ad). Charge for the compounding moat: shared team-wide Health Record (promoted
rules + fixtures + learned routing).
- Effort: XL. Risk: High.
- Aligns with Premise 1 (sell the moat) and solves "tired of free" (free is the
  funnel). Destination, not a starting line. Needs multi-user backend.

## Recommended Approach

**A first, then B, with C as the eventual pricing model.**

A is the only path that puts a real dollar number on the unverified core thesis this
week, using a paying customer's real data. High audit number → build B with
conviction and a reference customer. Low audit number → a month of wrong-loop
building avoided. B is where recurring revenue lives. C prices the moat once
adoption exists. Do not start at C.

## Open Questions

- What is the actual eliminable-class fraction of spend? (The whole bet. Unmeasured.)
- Of that fraction, what split is lint-catchable (eliminable, bucket 1) vs.
  tacit-judgment (routable-cheaper only, bucket 2)? Different economics (Premise 5).
- Do the founder's team logs / agent traces contain enough structure to bucket calls
  by "trivial vs. real thinking"? If not, what's the minimum instrumentation?
- Pricing: flat subscription, usage-based, or % of demonstrated savings?
- First language for B (the audit in A may be language-agnostic since it reads logs).

## Success Criteria

- **A:** One paying customer (or the founder's own org as customer zero) receives a
  retroactive audit with a credible eliminable-class fraction. Target signal: the
  number is clearly large (e.g. >40%) or clearly small (<15%). Either is a win; it
  resolves the bet.
- **B:** One honest, repeatable screenshot: "47 issues, 41 fixed locally for $0, 6
  escalated for $0.03, saved ~$0.40," from a real repo, reproducible run to run.
- **Revenue:** Money changes hands before the full live loop is built (paid audit).

## Distribution Plan

- **A:** Delivered as a service/CLI run on customer-exported logs. No host
  integration. Could even be a manual, founder-run service for the first few
  customers (concierge) before any tooling exists.
- **B:** Standalone CLI (`tokenclinic scan`), distributed via the language's package
  manager (npm/PyPI) + GitHub Releases. Fully self-controlled. No dependency on
  Claude Code / Cursor internals.
- Harness auto-switch integration is explicitly deferred to v2+ (Premise 4).
- CI/CD for the CLI: GitHub Actions building + publishing to the package registry on
  tagged release. Defer until B is real.

## Dependencies

- Access to real LLM call logs / agent traces with enough structure to bucket calls.
  This is the critical prerequisite for A. If logs are too thin, step zero is
  instrumenting one project for two weeks.
- Native tools per language (tsc/ruff/eslint) for B's triage. fff for retrieval.
  ast-grep for B/v2 structural rules.
